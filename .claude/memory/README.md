# Memory Bank 系統使用指南

## 📚 什麼是 Memory Bank？

Memory Bank 是一個**專案記憶持久化系統**，讓 AI 助手（如 Claude）能夠在不同的對話會話中記住專案的重要資訊、決策和上下文。

即使使用 `/clear` 清除對話，專案知識仍然保留在這些文件中。

---

## 📁 文件結構

```
.claude/
└── memory/
    ├── README.md           # 本文件 - 使用指南
    ├── project_overview.md # 專案總覽和基本資訊
    ├── architecture.md     # 技術架構和設計模式
    ├── features.md         # 功能清單（已實現/計劃中/已拒絕）
    ├── decisions.md        # 重要決策記錄和原因
    └── todos.md            # 待辦事項和任務追蹤
```

---

## 📖 各文件說明

### 1. project_overview.md
**用途**: 專案的「名片」

**包含內容**:
- 專案基本資訊（名稱、版本、目的）
- 技術棧清單
- 專案結構
- 快速開始指南
- Git 倉庫資訊

**何時更新**:
- 專案初始化時
- 技術棧變更時
- 目錄結構重大調整時
- 版本號變更時

---

### 2. architecture.md
**用途**: 技術架構文檔

**包含內容**:
- 組件架構圖
- 狀態管理方式
- 數據流向
- 關鍵功能模塊解釋
- 動畫和樣式架構
- 性能優化策略

**何時更新**:
- 架構重構時
- 添加新的設計模式時
- 狀態管理方式改變時
- 性能優化實施後

---

### 3. features.md
**用途**: 功能追蹤和規劃

**包含內容**:
- ✅ 已實現功能（詳細列表）
- 🚧 進行中的功能
- 📋 計劃中的功能
- ❌ 已拒絕的功能（含原因）
- 版本歷史

**何時更新**:
- 實現新功能時
- 功能狀態改變時
- 決定不實現某功能時
- 發布新版本時

---

### 4. decisions.md
**用途**: 決策記錄和知識傳承

**包含內容**:
- 技術選型決策（為什麼選 React？為什麼用 Vite？）
- 架構決策（為什麼用單一組件？）
- 功能決策（為什麼只有 3 個寶箱？）
- UI/UX 決策
- 待定決策
- 決策原則

**何時更新**:
- 做出重大技術決策時
- 拒絕某個方案時
- 改變原有決策時
- 有人質疑某個設計時

---

### 5. todos.md
**用途**: 任務管理和進度追蹤

**包含內容**:
- 🔥 緊急/高優先級任務
- 📌 中優先級任務
- 🔮 低優先級/未來功能
- 🐛 已知問題/Bug
- 📝 文檔待辦
- 🧪 測試待辦
- ✅ 完成的任務
- 📊 進度追蹤

**何時更新**:
- 添加新任務時
- 完成任務時
- 任務優先級變更時
- 發現 Bug 時
- 每週審查時

---

## 🔧 如何使用 Memory Bank

### 開始新對話時

1. **讀取記憶**（AI 助手應該做的）:
```bash
# AI 讀取相關的記憶文件
cat .claude/memory/project_overview.md
cat .claude/memory/todos.md
```

2. **快速了解**:
   - 專案是什麼？→ `project_overview.md`
   - 如何運作？→ `architecture.md`
   - 有什麼功能？→ `features.md`
   - 為什麼這樣做？→ `decisions.md`
   - 接下來做什麼？→ `todos.md`

### 開發新功能時

1. 檢查 `todos.md` 確認優先級
2. 閱讀 `architecture.md` 了解如何整合
3. 實現功能
4. 更新 `features.md` 標記為已完成
5. 如果有重要決策，記錄到 `decisions.md`

### 遇到問題時

1. 檢查 `todos.md` 的 Bug 區域
2. 查看 `decisions.md` 了解原始設計意圖
3. 參考 `architecture.md` 理解系統如何運作

### 定期維護

**每週**:
- 更新 `todos.md` 的進度
- 標記完成的任務
- 調整任務優先級

**每月**:
- 審查所有記憶文件
- 更新過時的資訊
- 歸檔完成的任務

**發布新版本時**:
- 更新 `project_overview.md` 版本號
- 在 `features.md` 添加版本歷史
- 記錄重大變更

## 🤖 Custom Instructions 建議

### 在 Claude 設定中添加以下指示：

```markdown
## 專案記憶系統（Memory Bank）

當處理此專案時，請遵循以下記憶管理原則：

### 開始工作時
1. 總是先檢查 `.claude/memory/` 目錄
2. 根據任務類型讀取相關記憶文件：
   - 了解專案 → project_overview.md
   - 理解架構 → architecture.md  
   - 查看功能 → features.md
   - 理解決策 → decisions.md
   - 查看任務 → todos.md

### 工作過程中
1. 當做出重要技術決策時，更新 `decisions.md`
2. 當實現新功能時，更新 `features.md`
3. 當完成任務時，更新 `todos.md`
4. 當架構變更時，更新 `architecture.md`

### 記憶文件維護
1. 保持文件簡潔但完整
2. 使用清晰的標題和結構
3. 記錄「為什麼」而不只是「是什麼」
4. 定期審查和更新過時資訊

### 原則
- **持久記憶優先**: 重要資訊應記錄在 Memory Bank 中
- **上下文感知**: 使用記憶文件快速理解專案狀態
- **知識傳承**: 讓未來的開發者（包括 AI）能快速上手
```

## 💡 使用技巧

### 技巧 1: 使用 `/memory` 命令
在與 AI 對話時，使用 `/memory` 讓 AI 快速回顧專案記憶。

### 技巧 2: 引用記憶文件
```
請參考 .claude/memory/architecture.md 來實現新功能
```

### 技巧 3: 更新提醒
在重大變更後提醒 AI 更新記憶：
```
請更新 features.md，標記 XXX 功能為已完成
```

### 技巧 4: 決策記錄
做重要決定時：
```
請在 decisions.md 記錄我們為什麼選擇方案 A 而不是方案 B
```

### 技巧 5: 定期審查
設定提醒，每週審查一次 `todos.md`

## 🎯 最佳實踐

### ✅ 好的做法

1. **及時更新**: 做出決策後立即記錄
2. **詳細記錄原因**: 不只記錄「做了什麼」，更要記錄「為什麼」
3. **結構化資訊**: 使用標題、列表、表格組織資訊
4. **保持同步**: 代碼變更時同步更新記憶文件
5. **版本控制**: 將 `.claude/memory/` 加入 git

### ❌ 避免的做法

1. **過度詳細**: 不要記錄每一個小細節
2. **忘記更新**: 完成任務後忘記更新 todos
3. **資訊重複**: 避免在多個文件重複相同資訊
4. **過時資訊**: 不刪除或更新過時的資訊
5. **缺乏結構**: 雜亂無章的記錄

使用 Memory Bank 系統後，您將獲得：
