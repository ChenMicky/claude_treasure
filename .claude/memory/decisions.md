# 重要決策記錄 - Treasure Hunt Game

## 技術選型決策

### 1. 為什麼選擇 React + TypeScript？

**決策日期**: 專案初始化時  
**決策者**: 專案團隊

**選擇原因**:
- ✅ **React**: 組件化架構，易於維護和擴展
- ✅ **TypeScript**: 提供類型安全，減少運行時錯誤
- ✅ **學習目的**: 這是教育專案，React 是最流行的前端框架
- ✅ **生態系統**: 豐富的第三方庫和工具支持

**替代方案考慮**:
- ❌ Vue.js - 也是不錯的選擇，但 React 更主流
- ❌ 原生 JavaScript - 缺乏組件化和狀態管理
- ❌ Angular - 對於這個小專案過於重量級

---

### 2. 為什麼選擇 Vite 而非 Create React App？

**決策日期**: 專案初始化時

**選擇原因**:
- ✅ **速度**: Vite 比 CRA 快得多（使用 ESBuild）
- ✅ **現代化**: 原生 ES 模塊支持
- ✅ **輕量**: 配置簡單，沒有多餘功能
- ✅ **HMR**: 熱模塊替換非常快速

**替代方案**:
- ❌ Create React App - 較慢，配置複雜
- ❌ Next.js - 對於靜態遊戲過度設計
- ❌ Webpack 手動配置 - 太複雜

---

### 3. 為什麼使用 Framer Motion 做動畫？

**決策日期**: 實現動畫功能時

**選擇原因**:
- ✅ **聲明式 API**: 與 React 完美配合
- ✅ **性能優秀**: 使用 GPU 加速
- ✅ **功能豐富**: 支持各種動畫效果
- ✅ **易於使用**: 學習曲線平緩

**替代方案**:
- ❌ CSS Animations - 功能有限，難以控制
- ❌ GSAP - 更強大但學習曲線陡峭
- ❌ React Spring - API 較複雜

---

### 4. 為什麼選擇 Tailwind CSS？

**決策日期**: 樣式系統選擇時

**選擇原因**:
- ✅ **快速開發**: 工具類直接在 JSX 中使用
- ✅ **一致性**: 設計系統內建
- ✅ **響應式**: 斷點系統簡單易用
- ✅ **優化**: 自動清除未使用的 CSS

**替代方案**:
- ❌ CSS Modules - 需要寫更多 CSS
- ❌ Styled Components - 運行時成本
- ❌ 原生 CSS - 難以維護

---

## 架構決策

### 5. 為什麼使用單一組件架構（App.tsx）？

**決策日期**: 架構設計時

**選擇原因**:
- ✅ **簡單性**: 遊戲邏輯簡單，不需要複雜分層
- ✅ **可讀性**: 所有邏輯在一個文件中，易於理解
- ✅ **性能**: 避免不必要的 props drilling
- ✅ **教育目的**: 便於學習者理解完整流程

**何時重構**:
- 當遊戲功能增加到 500+ 行時
- 當需要復用組件時
- 當狀態管理變複雜時

---

### 6. 為什麼不使用 Context API 或狀態管理庫？

**決策日期**: 狀態管理設計時

**選擇原因**:
- ✅ **簡單足夠**: useState 已滿足需求
- ✅ **避免過度設計**: 遊戲狀態簡單
- ✅ **性能**: 沒有不必要的重渲染問題

**何時考慮引入**:
- 多個深層嵌套組件需要共享狀態
- 實現複雜的排行榜或用戶系統
- 添加多人遊戲功能

---

## 功能決策

### 7. 為什麼只有 3 個寶箱？

**決策日期**: 遊戲設計時

**選擇原因**:
- ✅ **平衡性**: 3 個提供適當的選擇和挑戰
- ✅ **視覺**: 在不同螢幕尺寸上都能良好顯示
- ✅ **快速遊戲**: 每局遊戲時間短，適合休閒玩家
- ✅ **數學**: 1/3 機率容易理解

**未來計劃**:
- 考慮添加難度選項（3/5/7/10 個寶箱）

---

### 8. 為什麼分數是 +$100 / -$50？

**決策日期**: 計分系統設計時

**選擇原因**:
- ✅ **獎勵優先**: 正面反饋更多
- ✅ **心理學**: 鼓勵玩家繼續嘗試
- ✅ **簡單數字**: 易於計算和理解
- ✅ **遊戲性**: 即使遇到兩個骷髏，找到寶藏仍能回正

**計算邏輯**:
- 最佳情況: +$100 (直接找到)
- 中等情況: $0 (一個骷髏 + 寶藏)
- 最差情況: -$100 (兩個骷髏 + 寶藏)

---

### 9. 為什麼音效在點擊時才播放？

**決策日期**: 音效系統實現時

**選擇原因**:
- ✅ **用戶體驗**: 避免自動播放被瀏覽器阻止
- ✅ **反饋**: 與用戶操作直接關聯
- ✅ **控制**: 用戶可以控制是否靜音（瀏覽器設定）

**技術實現**:
```typescript
const audio = new Audio(soundFile)
audio.play().catch(error => {
  console.log('Audio playback failed:', error)
})
```

---

### 10. 為什麼實現 WIN/TIE/LOSE 顯示？

**決策日期**: 2026-04-04（最近添加）

**選擇原因**:
- ✅ **清晰反饋**: 玩家立即知道結果
- ✅ **視覺衝擊**: 大字體彩色顯示吸引注意
- ✅ **遊戲化**: 增加成就感和挫折感
- ✅ **用戶要求**: 根據反饋添加

**實現細節**:
- WIN: score > 0 (綠色)
- TIE: score === 0 (黃色)
- LOSE: score < 0 (紅色)

## UI/UX 決策

### 11. 為什麼選擇海盜主題？

**決策日期**: 設計階段

**選擇原因**:
- ✅ **寶藏遊戲契合**: 海盜 + 寶箱是經典組合
- ✅ **表情符號**: 🏴‍☠️ 💰 💀 視覺效果好
- ✅ **顏色方案**: 琥珀色/金色符合寶藏主題
- ✅ **普遍性**: 全年齡層都能理解

---

### 12. 為什麼使用表情符號而非圖標？

**決策日期**: UI 設計時

**選擇原因**:
- ✅ **簡單**: 不需要圖標庫或 SVG
- ✅ **通用**: 跨平台一致顯示
- ✅ **大小**: 不增加包大小
- ✅ **有趣**: 更輕鬆的視覺風格

---

### 13. 為什麼使用 180° 翻轉動畫？

**決策日期**: 動畫實現時

**選擇原因**:
- ✅ **視覺效果**: 類似真實翻牌效果
- ✅ **期待感**: 動畫過程增加懸念
- ✅ **時長適中**: 0.6 秒不會太長或太短
- ✅ **流暢性**: 配合 easeInOut 更自然

## 決策原則

### 核心原則
1. **簡單優先**: 選擇最簡單有效的解決方案
2. **教育目的**: 代碼要易於理解和學習
3. **性能**: 不犧牲用戶體驗
4. **可維護性**: 易於未來擴展

### 評估標準
- ✅ 是否滿足當前需求？
- ✅ 是否易於理解和維護？
- ✅ 是否有良好的社區支持？
- ✅ 是否符合專案規模？
